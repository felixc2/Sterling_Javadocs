Indicates how the OrderLines are dependent on the parent line. The DependencyShippingRule can only be specified for the parent line.
If DependencyShippingRule is specified on the parent line, it is defaulted to the same value on all the dependent lines. If DependencyShippingRule is passed on any of the dependent lines, the value passed is ignored.

Valid values: 01 - Ship together, 02 - Deliver together

If DependencyShippingRule value is 02 , a MergeNode must be specified. Order release created for these lines have a MergeNode value, so shipments are first made to the MergeNode, where they are consolidated for final shipment.

IntentionalBackorderflag (maximum length: 1) yfs_order_line.intentional_backorder
This flag indicates whether the order line should be backordered upon creation. If passed as 'Y', the line will be backordered even if inventory is available. If a reservation ID is passed on the order line and reservation exists for a current, past or future date, reservation will be consumed to block inventory for the backordered demand. In this scenario, demand created by the reservation ID is removed, and a backordered demand will be generated. No attempt is made to look for inventory if reservation is not available for the passed reservation ID. For draft orders, the reservation is consumed and the order line is created in created status. During confirmDraftOrder, this line goes in backordered status. This flag is ignored if reservation is mandatory. This flag should not be passed for bundle parent lines. If it is passed as 'Y' for bundle parent order line, the line is created in backorder status. It is not propagated to bundle components.

ChargeSequencenonNegativeInteger (total number of digits: 5) yfs_payment.charge_sequence
Used to identify the order to which charges are applied.
The Seller Organization can configure the payment types so that they are charged and refunded in a particular sequence. The charge sequence and refund sequence are independent of each other. A PaymentType with a ChargeSequence set to 0 is charged before another PaymentType with ChargeSequence set to 1 .
The Buyer can also specify the ChargeSequence of the PaymentMethod in the createOrder input xml. If a charge sequence is not specified by the Buyer, the Seller's ChargeSequence is followed.

If both the Buyer and the Seller supply ChargeSequence , charges are made in the order that the Seller specifies, but within a particular charge type, such as CreditCard, the Buyer's sequence is honored.

For example:
A seller specifies the ChargeSequence as 0 for CUSTOMER_ACCOUNT, 1 for CREDIT_CARD, 2 for OTHER.
A buyer specifies the ChargeSequence as 1 for MyBank-CreditCard, 2 for CUSTOMER_ACCOUNT, and 3 for MyOther-CreditCard.
The payment methods are charged in the following sequence:
CUSTOMER_ACCOUNT, MyBank-CreditCard, MyOther-CreditCard.
The Customer account is charged because that is the seller's preferred method, and the two credit cards are charged based on the order that the buyer specified.
